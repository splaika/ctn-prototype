# 治験届システム化 要件・考慮点 まとめ

本書は、これまでの検討（動的UIの試作 → Power Platform適合性 → Dataverseデータモデル＋PCF → 外字対応 → 順序番号 → 分担医師の異動管理 → SaaS型UX → Microsoft 365 Copilot／Agent 365 の方向性）を、**要件**と**考慮点**として一本化したもの。`CTN_システム化検討まとめ.md` を要件定義の形に拡張・統合した上位ドキュメントとして扱う。

## 0. 目的・スコープ・前提

### 0.1 目的
治験届（治験計画届／変更届／中止届／終了届／開発中止届）の作成・社内承認・提出を電子化し、PMDA規格に沿った XML/PDF を生成。データインテグリティ・監査証跡・職務分離・電子署名を担保する。

### 0.2 スコープ
- 対象：医薬品治験届を中核（医療機器・再生医療等製品は将来拡張）。
- 含む：届出作成、入力支援・検証、社内ワークフロー、XML/PDF生成、添付資料管理、監査。
- 含まない（初期）：PMDAゲートウェイへの自動送信（Phase 1は手動アップロード）。

### 0.3 段階導入
| | Phase 1 | Phase 2 | Phase 3（構想） |
|---|---|---|---|
| 主眼 | 社内ワークフロー電子化 | XML自動生成・提出経路 | エージェント支援・統治 |
| 実装 | Dataverse＋モデル駆動アプリ＋最小PCF | Azure Functions（XML/XSD）、提出連携 | Copilot Studio エージェント＋Agent 365 統治 |
| 検証区分 | 主にCat 4 | Cat 5（XML生成）追加 | Cat 5（エージェント）＋運用統制 |

### 0.4 前提
- プラットフォームは Microsoft Power Platform / Dataverse を中核とする。
- 規制：ER/ES指針、CSV（GAMP 5）、21 CFR Part 11、PMDA電子申請基準に準拠。
- 計算・検証の正本は常に**サーバー側（プラグイン／フロー）**に置き、UI（PCF）やエージェントはバイパス可能な補助層とみなす。

## 3. 重要トピック詳細

### 3.1 順序番号（2類型）
順序番号は治験薬専用ではなく、**届出XML内の繰り返し要素に振る通し番号**。性質で2類型に分ける。

- **突合キー型（治験薬）**：含量・剤形別に採番し、計画届→終了届で同じ番号のまま数量を突合。削除時も詰め直さない（再連番は突合を壊す）。
- **イベント行型（分担医師）**：追加・削除・氏名変更を「順序番号＋異動区分＋日付＋理由」で表現。届ごとに1から振り直す。同一人物が同じ届で追加・削除されれば2行になる。人の同一性は順序番号ではなく不変の医師IDで管理する。

参考シナリオ（分担医師、ダミーデータ）：

| 届 | 順序番号 | 異動区分 | 医師ID | 氏名（届出） | 理由 |
|---|:--:|:--:|---|---|---|
| 計画届 | 1/2 | 登録 | D001/D002 | 佐藤一郎／鈴木花子 | 初回登録 |
| 第1回変更 | 1/2 | 追加/削除 | D003/D001 | 田中次郎／佐藤一郎 | 体制強化／異動 |
| 第2回変更 | 1/2/3 | 追加/追加/削除 | D004/D005/D005 | 渡辺健一(原字渡邊)／中村桃子／中村桃子 | 体制強化／補強／短期異動 |
| 第3回変更 | 1 | 変更 | D002 | 鈴木花子→高橋花子 | 婚姻による氏名変更 |

### 3.2 外字（二段構え）
PMDA治験届は常用漢字（JIS第1・第2水準）のみ。外字・旧字は置換し、置換方法の説明資料を添付する必要がある。

- **原表記**：入力どおりUnicodeで保持（正確性・原本性／ALCOA+）。
- **届出用表記**：JIS第1・第2水準へ正規化した値。
- **流れ**：水準外/機種依存/IVS/PUAを検出 → 代替字を確定 → 置換マッピング記録 → 外字置換説明資料を自動生成。NFC正規化、半角/全角ルール、字数・バイト数制限も同層で検証。
- 正規化は標準縮退マップ＋社内辞書による**支援付き**（完全自動の1対1変換ではない）。真のユーザー外字（PUA＋専用外字フォント）はクラウドで字形再現不可のため運用禁止。

## 4. UI/UX要件

- 一度に1つのことに集中させ、認知負荷を抑える（情報過多を避ける）。
- 構造：上部メニューバー＋3ステップ（異動の登録 → 確認 → 提出）のSaaS型ワークフロー。
- **核心の分離**：利用者は「ロスター（現在の状態）を足す・抜く・直す」を自然に操作し、システムが順序番号イベント行を自動生成する。利用者に順序番号を意識させない。
- 詳細（順序番号・XML・医師ID・外字）は前面に出さず、確認ステップ等で開く。
- 提出は承認後のみ可（職務分離）。同一性・履歴は専用ビューで可視化。
- （任意）Copilotチャットペイン／エージェントによる自然言語アシスト（変更要約・ドラフト）を上乗せ層として提供できる。記録確定・提出は会話UIに行わせない。

## 5. アーキテクチャ

### 5.1 基本構成（Power Platform）
- データ＝Dataverse、UI＝モデル駆動アプリ＋必要箇所のみPCF、ワークフロー＝Power Automate＋ビジネスプロセスフロー、文書＝SharePoint、XML生成＝Azure Functions、可視化＝Power BI。
- 検証区分：データモデル・フォーム・フロー・ロール＝Cat 4／プラグイン・PCF・XML生成＝Cat 5。

### 5.2 UIの選択肢と方向性
1. **モデル駆動アプリ（基本）**：規制記録の入力。監査・権限と相性が良い。
2. **PCF**：本プロトタイプ並みの作り込みが要る箇所のみ（区分サジェスト・期限バナー・外字検証・完了度等）。Cat 5。
3. **Copilot/エージェント層（任意・Phase 2+）**：自然言語アシスト。M365 Copilotはチャットペインでの要約・操作に向かう方向。
4. **Agent 365 によるエージェント統治（構想・Phase 3）**：エージェントをID・権限・監査つきで登録・監視・統制するコントロールプレーン。

### 5.3 エージェント採用時の原則
決定的な中核（Dataverse＋検証済みプラグイン／フロー）を「ツール／データ」として残し、その上に用途を絞ったエージェント群（作成支援・変更要約・外字チェック・ドラフト）を被せる。エージェントは**提案・要約・オーケストレーションに徹し、記録の確定・検証・提出は自律的に行わせない**。最終確定は人間（HITL）＋決定的ゲート。Agent 365 を使う場合、各エージェントはEntra Agent ID・最小権限・Data & tools限定・Security & compliance・Activity監視を備える。

## 6. 考慮点（Considerations）

### 6.1 Dataverse実装の特性と留意点
- **採番**：標準オートナンバーはテナント全体で一意。順序番号（突合キー型／イベント行型）は同期プラグインで親スコープ採番し、同時実行制御を実装。
- **計算・集計**：数式列は単純導出向き、ロールアップは非同期（即時反映されない）。複雑判定（区分・30日・外字・採番）はプラグインで正本化。
- **日付**：規制日付は「日付のみ＋タイムゾーン非依存」（時差による日付ずれ防止）。
- **文字**：UTF-16で原字保持可。PMDAのバイト数・字数制限（例：255バイト、カナ全角50/半角100）はDataverseの文字数指定とは別にバイト換算で検証。サロゲートペアの字数計算に注意。
- **論理削除**：物理削除はせず statecode（有効/無効）。分担医師の「削除」はイベント行として残す。
- **キー**：治験成分記号を代替キー化し重複防止・突合に利用。
- **監査・セキュリティ**：テーブル監査を有効化し一般ユーザーが無効化・改変できないことをOQで確認。ロールで職務分離、提出後不変列はFLSで保護。
- **プラグイン**：同期で検証・採番。実行時間制限があり、重いXML生成はAzure Functions側へ。
- **PCF**：クライアント側でバイパス可。検証はUXに留め、必ずサーバー側に同等の正本を置く。
- **ALM**：ソリューションで管理、開発/検証/本番を分離、マネージドで配布（移行手順をIQ証跡に）。
- **月次更新**：Power Platformの更新に対する再検証方針を計画書に明記。

### 6.2 エージェント／Agent 365 採用時の留意点
- Agent 365 は「観察・統治・保護」のコントロールプレーンであり、Copilot Studio等で作ったエージェントを**中央統治する基盤**（開発基盤ではない）。Entra Agent ID・Conditional Access・最小権限・ライフサイクル、Purview監査（査察対応レベルのフォレンジック）・DLP・Insider Risk、Defenderによる脅威保護、利用テレメトリを備える。
- これらの統治機能は、規制系の要件（アクセス制御・監査・説明責任・定期レビュー）と噛み合う。タブ対応：Permissions＝アクセス制御・職務分離、Data & tools＝検証済みツール台帳・データリネージ、Security & compliance＝Part 11統制、Activity＝監視・継続的プロセス検証。
- **非決定性**：エージェントはLLM駆動で非決定的。記録の正本・検証・提出に使わない。検証は「呼ばれる側の決定的ツール」に対して行い、エージェントは意図された用途＋ガードレール＋監視＋HITLで縛る。
- **成熟度**：Agent 365 は新しく、月次更新の再検証負荷・プラットフォーム成熟度リスクがある。Microsoft自身も段階的アプローチ（発見・インベントリ → 低リスクのパイロット → ポリシー反復）を推奨。提出に直結する基幹ロジックを最初に乗せない。
- **DLP/ラベル**：規制データ保護に有効な反面、正当に治験データを扱うエージェントが過剰に止められないよう、ラベル・ポリシーを意図的に設計。
- エージェント層は Cat 5。低リスクの支援（変更要約・外字チェック）から段階導入する。

### 6.3 CSV／GxP・規制要件
- 「人間の最終承認」をリスク低減コントロールとして登録し、XML生成等の検証範囲を絞る論理をリスクアセスメントに残す。
- **OQ必須テスト例**：①承認前は提出不可 ②監査を無効化・改変できない ③職務分離（起案者≠承認者）の強制 ④電子署名が記録に紐づく ⑤XMLがXSDで妥当 ⑥外字が検出・置換・記録され説明資料が生成される ⑦治験薬の順序番号が終了届で保持され数量が突合できる ⑧分担医師の異動がイベント行（順序番号＋異動区分＋日付＋理由）で正しく生成される ⑨（エージェント採用時）エージェントの権限が最小化され、記録確定・提出を自律実行しないこと。

### 6.4 非機能要件（抜粋）
- データインテグリティ（ALCOA+）、可用性、性能（大量の施設・医師・届出への対応）、国際化（日本語・外字・国際共同治験）、変更管理（プラットフォーム更新への追随）、バックアップ・保持。

## 8. 用語
- **CTN**：治験届（Clinical Trial Notification）。
- **30日調査**：初回計画届等で治験開始30日前までの提出を要する区分1の届出。
- **順序番号（突合キー型／イベント行型）**：本書3.1参照。
- **外字／二段構え**：本書3.2参照。
- **HITL**：Human-in-the-loop（人間の確認・承認を介在させる統制）。
- **Agent 365**：Microsoftのエージェント統治コントロールプレーン。
- **PCF**：Power Apps Component Framework（カスタムUIコンポーネント）。
