CSV 最小統制 & ライセンス設計 検討・設計

治験届システムに「本当に必要な CSV の重さ」と、Power Platform 実装時のライセンス実額

2026-06-26 設計討議サマリ / ©Seven to One / 社内共有用・noindex

結論の筋：Cat4/5（GAMP）→ 人手承認のクレジット（CSA）→ 治験届 XML は既製ツールで作るのが業界標準 → 最小 CSV 統制セット（ログイン帰属＋生成物一致記録）→ Power Platform 実装（Entra＋Dataverse 監査）→ ライセンスは年 ¥10–15 万・1 アプリで足りる。

01Cat 4 / Cat 5 は GAMP 5 のソフトカテゴリ

このプロトタイプの「Cat 4 / Cat 5」は治験届の届出区分ではなく、GAMP 5 のソフトウェアカテゴリ。コンポーネントを「どう作ったか」で分類し、検証（CSV）の重さを決める枠組み。

  • Cat 4＝構成（設定）で作る製品（Dataverse／フォーム／Power Automate／SharePoint）→ 検証は構成仕様書＋OQ で軽い。
  • Cat 5＝自社開発コード（Azure Functions の XSD 準拠 XML 生成等）→ 詳細設計＋単体テスト＋CSV 一式で重い。

検証の重さ＝コンポーネントの作り方（カテゴリ）で決まる＝リスクベース配分。Cat 2 は GAMP 5 で廃止。

02人手承認の「クレジット」で Cat 5 負荷は落とせる（CSA 的発想）

人がレビュー・承認する＝検出コントロールが 1 枚噛む。ならば同じリスクをコードの網羅検証で二重に潰す必要はない（＝検証対象が"移動"する、CSA: Computer Software Assurance の考え方）。

クレジットの成立条件：① その統制が本当にリスクを検出できる ② 統制自体が確実に効く（飛ばせない・記録が残る）。この 2 点を検証計画に根拠として書いて初めて、対応するソフト検証を正当に削れる。

PDF は人が読めるので◎。ただし XML は人が読まない部分（外字・符号化）が△ ＝ ここだけ残課題（後述の入力時バリデーションで入口に潰す）。

03治験届 XML は「既製ツールで作る」のが業界標準

JPMA「治験届の XML ファイル作成の手引き 第4版」別添5（2024-10 改訂）に XML データ作成システム提供会社一覧＝6 社が公式掲載されている。＝「XML は買って作る／CTMS の一機能で吐く」が業界の共通認識で、自前で生成エンジンを組む前提は業界にない。

04最小 CSV 統制セット（届出 prep システムに必要十分）

治験届は届出制（実質審査付き）＋ PMDA 30 日調査＋人手ゲート多重のリスク階層。Cat5 のフル CSV ではなく、人手承認をクレジットして以下で「査察に説明可能」に成立する。

統制担保するものALCOA+
ログイン／アクセス帰属（Entra ID）誰が入力・確認・提出したか辿れるAttributable
生成物の一致記録（承認済み原データ＝XML＝PDF、差分なし）提出物が承認内容と一致＝中身が正しいAccurate / Original
その記録が日時・担当者付きで残るいつ誰が確認したか後から証明できるContemporaneous / Enduring

治験届は令和2年8月改訂で社印捺印すら廃止 → 電子署名は必須要件でなく「帰属ある承認＋記録」で代替可。

本当に成立させる 3 条件

  1. 差分は"承認済み原本"基準で比較（PDF-XML 相互一致だけでは不足＝両方同じ間違いをすり抜ける）。
  2. 転記ステップが唯一の誤り混入点 → Excel を廃止して直接入力にすれば発生源ごと消える。
  3. 検証版＝提出版（差し替え防止・版固定）。

05実装（Power Platform）

  • Entra ID = 認証・アクセス制御（認証エンジンは Microsoft 持ち＝インフラ Cat1、検証は設定のみ＝Cat4 的に軽い）。ただし「誰が承認したか」は Entra に残らない → Dataverse 側で記録する。
  • 承認記録は届出レコード自体に刻む（承認者・日時・対象版スナップショット/版ID）＋そのテーブルを Dataverse 監査 ON（管理者管理の追記型・利用者は消せない・保持年数設定可＝tamper-evident）。co-located＋retained＋version-pinned＋complete を同時充足。
  • Power Automate の承認履歴"だけ"は不可：① 実行履歴は既定 28 日で削除（治験は年単位保管）② TTL で完全削除可・高負荷で欠落しうる＝改ざん耐性／全件性が監査証跡に不足 ③ どの版を承認したかに紐づかず別ストア。→ PA は"承認を回す道具"、原本は Dataverse レコード＋監査。
  • 入力画面化（Excel 廃止）の効き：転記ミス消滅＋入力時バリデーション強制（外字検出／文字数／コード辞書）＝人が読めない XML 部分の残課題も入口で潰せる。

06ライセンス試算2026 価格・約¥155/$

  • Power Apps Premium $20/人月（固定・割当）／ PAYG $10・その月にアプリを開いた実ユーザー・アプリ単位（使った月だけ・Azure 経由）／ Per App $5 は 2026-01 新規終了。
  • Dataverse アプリは閲覧でも有償エンタイトルメント必須・無料閲覧枠なし。Power BI 無料閲覧は F64+ 容量（~$5,257/月）のみ＝この規模は非現実的 → 見るだけの人は SharePoint／エクスポートで無料配信。
  • モデル駆動アプリのネイティブ・ダッシュボードは Power Apps 権利に含む＝Power BI 不要（内部運用監視はこれで足りる）。

071 アプリで全スコープを通せる

アプリ分割の 4 条件（別利用者層／別 UX・デバイス／独立ライフサイクル／商用パッケージング）は CTN で全否。同じ RA チーム・共有マスタ（治験成分 1:N）・同一ワークフロー＝1 つの入れ物に入れたい構造。

CTN（1 アプリ） ├─ 届出：計画届 / 変更届 / 中止 / 終了 / 開発中止 ├─ マスタ：施設 / 医師 / 治験成分 / IRB └─ ダッシュボード：ステータス / 承認待ち / 期限アラート
  • Phase 2（残り 4 届出＋AI 判定）・Phase 3（照会対応／PMDA Gateway 連携／eCTD v4 出力）も テーブル追加・裏側連携＝同一アプリで伸ばせる → PAYG は 1 ユーザー＝$10/月のまま。
  • アンチパターン：届出種類ごとにアプリ分割（PAYG 倍々・メリットなし）。種類は"アプリ"でなく"テーブル/フォーム"で分ける。
  • 商用の出し分けはセキュリティロール／機能フラグで（アプリ分割不要）。

08残課題

  1. 外字／文字コード変換の入口バリデーション仕様の詳細化。
  2. 既製 XML ツール（TRI/日立/HORIZON 等）連携 vs 自前生成の最終判断。
  3. 実人数・年間届出件数・為替でのライセンス試算ピン留め。
