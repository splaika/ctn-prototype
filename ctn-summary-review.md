# CTNシステム化 検討まとめ ─ レビュー

対象: `ctn-systemization-summary.html`（治験届システム化 検討まとめ・チーム向け解説版／©Seven to One）
観点: 整合性・抜け漏れ・リスク
根拠: 「治験届の XML ファイル作成の手引き 第4版」（日本製薬工業協会, 2021/02）本文、および本リポジトリの UXモック/マニュアル・XSDマッピング表たたき台（`outputs/ctn-xsd-mapping/`）と突合。

---

## 0. 総評

設計の骨格は妥当で、**手引き本文とよく整合**しています。とくに「治験成分＝親／各届＝子」のデータモデル、順序番号の2類型、外字の二段構え、Dataverse のハマりどころ、CSV/GxP の「HITLをコントロールとして登録する」という整理は、いずれも実装に耐える解像度です。

一方で、**(1) 手引きの確定値（STATUS の実体値・3.8 例外リスト）への落とし込み**、**(2) これまでの議論で合意した範囲（既存データのXML→マスタ化、文字数超過時の添付スピル、PDF/提出様式の位置づけ）の反映**、**(3) 公式XSDとの最終突合**——の3点が未反映です。いずれも設計を覆すものではなく、**「決定表に書き足す」レベルの追補**で閉じます。

優先度つきで以下に整理します。

---

## 1. 強み（このまま進めてよい点）

- **Cat4/Cat5 の切り分け**（Section 02）— XML生成・プラグイン・PCF のみ Cat5、他は設定。工数とCSV負荷の見積り軸として正確。
- **データモデル**（Section 03）— 治験成分を親にすることで、届出回数の通算・30日調査の起算・初回/変更の親子が自然に表現できる。妥当。
- **順序番号2類型**（Section 04）— 手引きの `SERIALNO1`/`SERIALNO2`（STATUS=`NONE`/`APPEND`/`DELETE`）の挙動と一致。突合キー型＝欠番許容・継承、イベント行型＝届ごと再採番＋不変医師ID、という分離は正しい。
- **外字二段構え**（Section 05）— 原表記/届出用表記＋GaijiMap＋置換説明資料は、手引きの「JIS第1・2水準のみ／置換時は説明資料添付」と整合。PUA禁止・IVS寄せの運用判断も妥当。
- **Dataverse留意点**（Section 07）— オートナンバー非対応、バイト数≠文字数、TZ日付ずれ、ロールアップ非同期、物理削除禁止、PCF迂回、ALM月次更新。いずれも実運用で必ず当たる real な論点を先回りできている。
- **CSV/GxP**（Section 08）— 「人の最終承認をコントロールとして登録」＋OQ8項目。⑥〜⑧が本システム固有設計の検証に落ちている点が良い。

---

## 2. 要確認・抜け漏れ（優先度順）

### [高] 2-1. STATUS 実体値が資料の用語と未接続
Section 04 の表は異動区分を「登録／追加／削除／変更」と日本語で表現しているが、**XMLの `STATUS` 属性の実体値は `NONE / UPDATE / APPEND / DELETE`**（手引き第4・5章で確認。"NEW" ではなく **`APPEND`**）。
→ 決定表に「業務区分 → STATUS属性値」の対応を明記する。目安: 初回登録/追加=`APPEND`、削除=`DELETE`、変更=`UPDATE`、変更なし/対象外=`NONE`。OQ⑧は**実体値まで**アサートすべき。
（対応表は `outputs/ctn-xsd-mapping/` に収録済み。）

### [高] 2-2. 3.8「STATUS=NONE 固定」項目リストが未収録
手引き 3.8 は、前回提出と内容が異なっても `STATUS` を変えない項目を定めている。これを持たないと「差分=変更」と誤判定する。
→ 「変更追跡する／常にNONE」のフラグを決定表に1列追加（未決事項①に統合可）。

### [高] 2-3. 既存データの XML→マスタ化（初期移行）が設計に未記載
過去の提出XMLは既に公式スキーマ準拠の構造化データのため、**AI抽出不要でパース→名寄せによりマスタを自動構築**できる（既存CTN対応の最短経路）。Section 09 の未決にもデータモデルにも明示がない。
→ Phase 1 の「初期データ移行（過去XML一括ETL）」として明記。医師・医療機関・薬剤・組織の dedup 方針を添える。

### [中] 2-4. 文字数超過時の添付スピル（3.10）がフロー/OQに未反映
長文項目（製造方法・治験計画概要 等）が `maxLength` を超えると検証エラー→提出不可。手引き 3.10 は「（添付資料名）のとおり」を入れて本体を添付に逃がす運用を規定。
→ 入力ガード（上限突合→スピル提案→プレースホルダ挿入→添付登録）を追加。OQに1項目追加を推奨。

### [中] 2-5. PDF生成／提出様式の位置づけが未整理
資料は提出をXML前提で記述しているが、届書本体・別添1（出力イメージ）のPDFの扱いが不明。
→ 「電子的提出の正本はXML、PDFは確認/控え」か「公式様式PDFを忠実再現して提出物にする」かを確定。前者なら Word テンプレ変換で軽く済む。

### [中] 2-6. コード参照マスタの整備が未決に未記載
大学番号（手引き第6章＝別添4）、剤形コード（日本薬局方）、薬効分類番号、投与経路コードの参照表整備が、外字辞書（未決③）と違い項目化されていない。
→ システム管理レイヤーの「コード・選択肢管理」に初期取り込み＋改定追従を1項目化。

### [低] 2-7. 30日調査の起算ルールの精緻化
親に 30日判定を持つ方針は良いが、変更イベント→起算日→期限→通知のルール詳細が未記述。既存UXモックの「期限・通知ルール」と接続して具体化を。

## 3. リスク

| リスク | 内容 | 緩和 |
|---|---|---|
| 採番の同時実行衝突 | 届スコープ採番をプラグインで実装（資料も認識済） | 一意制約＋楽観ロック＋リトライ。負荷テストをOQ化 |
| 公式XSDの版追従 | `iykckn_all_v3_0_0.xsd` 改定時にマッピング再検証が必要 | ALM-SOPに「XSD改定時の回帰テスト」を明記。スキーマ版を構成管理 |
| 要素名の表記ゆれ | 本たたき台はPDF抽出由来。`MUNUFACTURERIMPORTERCODE`、`COMB_FOREIGN_SPOMSPR_REP_NAME_FRGNLNG` 等、手引きのtypoか抽出ゆれか不明な要素名がある | **確定はPMDA配布の公式XSD（正式な要素名・出現順・型・桁・コード値）と必ず突合**。本たたき台はFS作成の出発点として扱う |
| 要素出現順 | XSDは子要素順を強制。手引きは「VARIABLELABEL等は値より前」と規定 | 生成器は決定表の順序キーに従って出力。検証は実XSDで担保 |

## 4. 推奨アクション

1. **未決事項①（XSDマッピング表）に本たたき台を充当**し、(a) STATUS実体値列、(b) 3.8 NONE固定フラグ、(c) 公式XSDとの突合（要素名・順序・型・桁）、を追記して FS 化する。
2. **Phase 1 に「過去XML→マスタ初期移行」を追加**（既存CTN対応の本線）。
3. **入力ガード（3.10 添付スピル／バイト数／NFC・全半角）と PDF位置づけ**を仕様に追記。
4. **コード参照マスタ**（大学番号・剤形・薬効分類・投与経路）の取り込み計画を未決に追加。
