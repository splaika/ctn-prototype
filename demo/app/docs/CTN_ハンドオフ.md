# CTN Suite デモシステム構築 ― Claude Code ハンドオフ

このドキュメントは、Claude.ai上での設計検討（Dataverseデータモデル・サーバーロジック・UI方針・XML生成方式）の結論を、**Claude Codeが前提知識ゼロで動くデモシステムを構築できる**ように引き渡すものである。同梱の `ctn-schema.json` が定義の単一ソース（Excel構築定義ブックと同一内容の機械可読版）。

---

## 1. デモの目的 ― 何を証明するためのデモか

本番はMicrosoft Power Platform（Dataverse＋モデル駆動アプリ＋プラグイン）で構築する。このデモはPower Platformを使わず、**設計の中核ロジックとUXが成立することをローカルWebアプリで証明する**ためのものである。デモが示すべき価値は次の4点。

1. **転記の排除** ― マスタとシリーズ親からの自動継承により、届出ごとの入力は差分だけになること
2. **二層検証** ― クライアントは提案（推奨・即時フィードバック）のみ、確定はサーバー側という構造が成立すること
3. **順序番号2類型の自動化** ― 突合キー型（薬）とイベント行型（医師）を利用者に意識させずに正しく採番できること
4. **XML自動生成** ― リレーションの木をたどってXSD構造のXMLを生成し、機械検証できること

逆に、デモで示さなくてよいもの：Entra ID認証（ユーザー切替ドロップダウンで模擬）、電子署名（記録のみ）、SharePoint（ローカル保存で代替）、PMDAゲートウェイ連携、監査ログの改ざん防止性（記録と閲覧ができれば十分）。

## 2. アーキテクチャ ― デモでも「サーバー正本」を再現する

デモであっても、**フロントとバックエンドを分離すること**。これは本番アーキテクチャの核心（PCF/ビジネスルール＝入力支援、プラグイン＝正本）をデモで体感させるためであり、単一SPAに全ロジックを置くと二層検証のデモにならない。

- **バックエンド**：Node.js（Fastify または Express）＋ SQLite。Dataverseの役割を演じる。`ctn-schema.json` の `tables` / `relations` からテーブルを生成し、`serverLogic` の各項目をAPI層のミドルウェア／サービスとして実装する。採番・区分再計算・職務分離・提出ゲート・外字検出・バイト数検証は**必ずサーバー側で実施**し、クライアントの値を信頼しない。
- **フロントエンド**：React＋Vite＋TypeScript。モデル駆動アプリ＋PCFの役割を演じる。既存プロトタイプ `ctn-ux-react-standalone.html`（リポジトリ内、スタンドアロンReact UX）のコンポーネント・画面構成を可能な範囲で流用してよい。
- **XML生成**：バックエンドのエンドポイント `POST /api/notifications/:id/generate-xml` として実装（本番のCustom API「GenerateCtnXml」に相当）。生成→XSD検証→ファイル保存→レスポンスでプレビュー返却、の順。XSD検証は `xmllint-wasm` か `libxmljs2` を使用。

**XSDについての重要な注意**：公式XSD（iykckn_all_v3_0_0.xsd）はリポジトリに含まれていない。デモでは `ctn-xsd-mapping.html` と `ctn-dataverse-columns.html` の要素定義から**簡略サブセットXSDを自作**して同梱し、READMEに「デモ用サブセットであり公式XSDではない」と明記すること。公式XSDが後日提供されたら差し替えられるよう、XSDファイルパスは設定値にしておく。

## 3. データモデル ― ctn-schema.json の読み方

- `tables[]`：14テーブル・154列。`columns[].inputUi` が「その項目を誰がどう入力するか」（フォーム手入力／親から自動／プラグイン採番／数式列など）。**「自動」系の列に入力UIを作らないこと**（転記排除のデモ価値が消える）。
- `columns[].requiredByType`：「計/変/中/終/開」の順の必須記号。フォームの動的必須・表示切替はこれと `requiredMatrix` から駆動する（ハードコード禁止）。
- `columns[].status` が「要確認」の19列：設計上の未確定箇所。デモでは備考欄の仕様どおり仮実装してよいが、**UIに「要確認」バッジを出す**こと（デモを見た人が本番仕様と誤解しないため）。
- `choices[]`：選択肢セット。整数値もこのまま使用。
- `relations[]`：ルックアップ構造。SQLiteでは外部キー＋`cascade` 方針（Restrict＝削除拒否）で表現。
- `serverLogic[]`：16件のサーバー側ロジック。実装の受入基準は第5節のデモシナリオで担保する。

## 4. 特に間違えやすい4つのロジック仕様

**(1) 順序番号は2類型を絶対に混ぜない。**
- 突合キー型（治験使用薬）：シリーズ内で採番し、計画届→終了届まで同一の薬は同一番号。届をまたいでコピーされる。削除されても欠番のまま、再利用・再付番は禁止。
- イベント行型（医師）：届ごと・イベント単位で採番。同一人物が「追加」と「削除」で別行・別番号になりうる。
- 施設別数量：治験使用薬の番号を**継承**する（独自採番しない）。

**(2) 医師はロスター操作で編集し、イベント行はシステムが作る。**
利用者UIは「現在の分担医師ロスター」への足す・抜く・直すのみ。保存時にサーバーが異動区分（追加/削除/変更）・異動日付・理由つきのイベント行を生成する。物理削除は行わない。医師の同一性は医師マスタの不変ID（改名しても不変）。XMLではイベント行の異動区分をSTATUS属性（追加→APPEND、削除→DELETE、継続→NONE）へ写像する。「変更」の表現は要確認事項なので、デモでは削除＋追加の2行に展開する実装とし、その旨をコード内コメントとREADMEに残す。

**(3) 外字は二段構え＋医師単位の確認履歴。**
氏名は原表記／届出用表記のペアで保持。サーバーがJIS第1・2水準外を検出したら、フロントに確認ダイアログを出させ、確認結果（元字・コードポイント・代替字・確認者・日時）をGaijiMapへ記録。**確認済みの医師には再度ダイアログを出さない**。デモの縮退マップは代表例（髙→高、﨑→崎、濵→濱→浜 など10件程度）の同梱JSONでよい。

**(4) XML生成は主従区分・医師区分・異動区分で出力先が分岐する。**
主たる被験薬（1届1行をサーバーで強制）→届ルート直下の要素へ。その他治験使用薬→INFOCOMBINATION（SERIALNO1）へ。責任医師→INFOINVESTIGATOR、分担医師→INFOSUBINVESTIGATOR（いずれもSERIALNO2）。数量→INFOQUANTITIESINVESTPRODUCT（SERIALNO2、薬名称は治験使用薬から文字列展開）。単票項目はSTATUS=UPDATE型、繰り返し行はADD型。

## 5. 受入基準 ― このデモシナリオが通ること

デモは以下を1本の台本として通しで実行できること。シードデータ（シリーズABC-123、医師マスタ5名うち1名外字含み、機関マスタ2件、IRB 1件、届出者1件、ユーザー4名：起案者A・レビューアB・承認者C・薬事D）を `seed/` に同梱する。

1. **シリーズ作成**：成分記号「ABC-123」で作成。21桁・「&」含み・全角混じりはサーバーでエラーになること（フロントの即時表示も出るが、APIを直接叩いてもエラーになることをテストで示す）。
2. **計画届**：種別選択で必須項目が切り替わる。薬2行（主たる1・その他1）に#1・#2が自動採番。施設1件をマスタから選択、責任医師1名＋分担医師2名をロスター追加（うち1名で外字ダイアログ→GaijiMap記録）。数量マトリクスに予定交付を入力。届出区分の推奨=1が理由つきで表示され、期限バナーが開始予定日−30日を示す。
3. **承認と職務分離**：起案者Aが自分で承認しようとするとサーバーが拒否。承認者Cで承認。承認前に提出ボタンが無効であること。
4. **XML生成**：確認ステップで生成。ルート/INFOCOMBINATION/SERIALNO/STATUSが第4節どおりに出力され、同梱XSDの検証をパスし、プレビュー表示とファイル保存ができる。
5. **変更届**：分担医師1名追加・1名削除→イベント行が自動生成され、確認ステップで初めて順序番号が提示される。区分推奨が変更箇所から算出される。12か月バッチ対象（分担医師の異動のみ）の保留件数が表示される。
6. **終了届**：交付〜廃棄の数量列が必須化される。薬の順序番号#1・#2が計画届と一致している（突合キーの不変性）。終了届の提出で保留カウンタがクリアされる。
7. **監査**：上記すべての操作について「誰が・いつ・どのレコードの・何を・どう変えたか」が監査ログ画面で閲覧できる。

各基準に対応する自動テスト（バックエンドはユニット＋API、採番・区分・外字・職務分離・XML生成は必須）を用意すること。XML生成ロジックは**ホスト非依存の純粋関数／クラスに分離**する（本番でDataverseプラグインに載せ替えるため）。

## 6. リポジトリと参照資料

- リポジトリ：`splaika/ctn-prototype`（public）。デモは `demo/` ディレクトリ配下に自己完結で作成（ルート直下の既存HTML群は変更しない）。
- 参照：`ctn-dataverse-columns.html`（XSD由来の列定義）、`ctn-xsd-mapping.html`（マッピング）、`ctn-ux-react-standalone.html`（UI流用元）、`ctn-deadline-rules.html`（期限ルール）、`ctn-code-tables.html`（コード表）。
- README必須記載：デモの位置づけ（Power Platform本番の模擬であること）、簡略XSDの注意、要確認19件の一覧、起動手順（`npm install && npm run dev` 一発を目標）。

## 付記：この会話で確定・変更になった設計判断（本体ドキュメントへの反映待ち）

1. **XML生成のホストをAzure FunctionsからDataverseプラグイン（Custom API「GenerateCtnXml」）に変更**。理由：CTN 1件のXML生成は軽量でサンドボックス制限に達しない／Azure側インフラ・認証・DLP・IQ拡張が不要になる／ロジックをライブラリ分離すれば将来Functionsへ載せ替え可能。Functionsが必要になるのはゲートウェイ自動連携・バッチ・高度監視が要件化した時点。→ `CTN_Dataverse_PCF設計.md` 第10節と `CTN_PowerPlatform_CSV計画.md` 該当箇所の改訂が残タスク。
2. **構築定義の単一ソース化**：`CTN_Dataverse構築定義ブック.xlsx`（人間用）と `ctn-schema.json`（機械用）は同一スクリプトから生成。今後の定義変更はスクリプト側を正として両方を再生成する。
3. **要確認19件**（Excelの黄色セル）は未確定。デモでは仮実装可、本番設計の確定には手引き2024年3月版との突合が必要。
