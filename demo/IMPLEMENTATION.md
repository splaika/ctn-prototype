# CTN Suite 実装まとめ（治験届管理システム・本番想定UIデモ）

治験届（CTN: Clinical Trial Notification）の作成・変更・提出を管理する、**本番（Power Platform）を意識したUIデモ**の設計・実装記録です。会話を通じた要件の変遷と、最終的な実装内容を言語化しています。

- **公開デモ**: https://splaika.github.io/ctn-prototype/demo/
- **スタック**: React 18 + TypeScript + Vite（Power Apps Code App 構成）
- **単一ソース**: [`ctn-schema.json`](app/src/ctn/ctn-schema.json)（14テーブル / 154列 / 16選択肢 / リレーション16 / サーバーロジック16）

---

## 1. 背景と目的

- **CTN** は、医薬品の治験計画・変更・中止・終了・開発中止を PMDA へ届け出る業務。添付の設計ブック（`ctn-schema.json`）とハンドオフ資料を単一ソースとして受領。
- 本番は Microsoft **Power Platform**（Dataverse ＋ モデル駆動アプリ ＋ プラグイン）で構築する想定。本デモは Power Platform を使わず、**設計の中核ロジックとUXがローカルWebアプリで成立すること**を先行検証するもの。
- クライアント（PCF／ビジネスルール相当）は**提案・即時表示のみ**、**確定はサーバー側**という二層構造を、純粋関数（`logic.ts` / `xml.ts`）とメモリ内リポジトリで再現。データ層は mock ⇄ Dataverse で差し替え可能（UIコードは不変）。

## 3. アーキテクチャ

```
demo/app/src/
  App.tsx                 シェル（画面遷移・ユーザー切替・言語・テーマ・トースト）
  i18n.ts / PowerProvider Power Apps ホスト待ち（Dataverseモード）
  index.css               デザインシステム（標準＝navy/blue、モダン＝light/indigo）
  ctn/
    ctn-schema.json       単一ソース
    schema.ts             スキーマローダー（choices / requiredByType / 要確認 抽出）
    types.ts              ドメインモデル
    refData.ts            choice値定数・利用者・外字マップ・日付ヘルパ・暫定選択肢
    logic.ts / xml.ts     サーバー正本ロジック（純粋関数）＋ XML生成
    derive.ts             派生（アラート・リマインダ・集計）
    rules.ts              ロジカルチェック設定（期限・アラートのルール）
    data/                 repository（mock/dataverse切替）・mockRepository・seed
    components/           Sidebar / Dashboard / NotificationList / NotificationDetail
                          / CreateWizard / MasterView / GaijiDialog / XmlPreview
                          / ImportView / SettingsView / AuditView / common
```

- **スキーマ駆動**: テーブル・列・選択肢・**動的必須切替**（`requiredByType`/`requiredMatrix`）・「要確認」バッジは `ctn-schema.json` から取得（ハードコード禁止）。
- **リポジトリ層**: コンポーネントは `CtnRepository` インターフェースのみに依存。`VITE_USE_DATAVERSE` の1変数で mock ⇄ Dataverse を切替。書き込みは監査ログを自動付随。
- **i18n**: EN/JA 切替（`t(en, ja)`）。既定は JA。

## 4. 画面と機能

- **ダッシュボード** — KPI（届出総数・提出済み・進行中・アラート・リマインダ）、アラート（薄ピンク）/リマインダ（薄黄）カード、全治験届の一覧。
- **治験届一覧** — 種別・状態・検索で絞り込み。列は**ドラッグで並び替え可能**（保存）。既定順：治験成分記号／治験計画書番号／届出種別／回数／届出区分／ステータス／提出期限／医療機関数。
- **治験届 詳細** — 縦長一覧を**横並びタブ**（基本情報／治験計画概要／治験使用薬／実施医療機関／参照・添付・照会）に分割。エクセル全項目を入力可（治験使用薬は展開式カードで20項目、施設はSMO等、参照治験届出）。ワークフロー進捗（作成中→レビュー中→承認済み→提出済み）、提出期限バナー、届出区分の推奨、XMLプレビュー。**詳細表示中は左メニューが「治験届一覧」に連動**。
- **新規届作成ウィザード** — 既存/新規シリーズ選択（成分記号の即時検証）。新規シリーズは計画届で固定のため種別選択ステップを省略。
- **マスタ管理** — 医療機関／医師／CRC／IRB／治験届出者。登録ポップアップはエクセル全項目。**論理削除/復活**（物理削除しない）。医師登録時の**外字検出ダイアログ→確認履歴記録**。医師/CRC一覧に所属医療機関名を表示。
- **データ取り込み** — PDF/XML のドラッグ＆ドロップ。CTN XML を解析し**ドラフトの届として取り込み**（詳細画面のXMLをDL→再取り込みのラウンドトリップ可）。「サンプルデータを取り込む」で動作確認可。
- **ロジカルチェック設定** — 提出期限オフセット・アラート閾値・各アラート/リマインダの有効化。**薬事担当（管理者）ロールのみ編集可**、他ロールは閲覧のみ。設定は**全社（システム全体）適用**（`computeDeadline` / `deriveAlerts` に反映）。
- **監査ログ** — 操作・期間（From/To）・検索でフィルタ。全操作を記録。
- **ユーザー切替 / 表示切替** — 右上で操作ユーザー（職務分離のデモ）、**デザインテーマ（標準／モダン）**、**ダーク／ライトモード**を切替。左メニューは**折りたたみ可能**（折りたたむとアイコンのみ表示）。いずれも `localStorage` に保持。

## 5. サーバー正本ロジック（16件・純粋関数）

| ロジック | 実装 |
| --- | --- |
| 30日調査対象の判定 / 提出期限の算定 | `is30DayReview` / `computeDeadline` |
| 届出区分の推奨・確定（1＞2＞3） | `recommendKubun` |
| 順序番号採番 **突合キー型**（薬・シリーズ内不変） | `nextStudyDrugSerial` |
| 順序番号採番 **イベント行型**（医師・届内単位） | `nextInvestigatorSerial` |
| 治験成分記号の検証 | `validateCompoundCode` |
| 外字検出・正規化・記録 | `detectGaiji` / `normalizeGaiji` |
| バイト数検証 | `checkByteLimit` |
| 職務分離の強制 / 提出ゲート | `canApprove` / `canSubmit` |
| 医師ロスター差分→イベント行 | `diffRoster` |
| XML生成・XSD検証（分岐：主従/医師/異動区分） | `generateCtnXml` / `validateAgainstSubset` |

`npm test`（Vitest 34件）で採番2類型・区分・外字・職務分離・XML分岐を自動検証。

## 6. データモデルとエクセル整合

- 添付エクセルの **選択肢16 / 列154（種別別必須含む）/ リレーション16** が現行 `ctn-schema.json` と**完全一致**（差分ゼロ）を都度スクリプトで確認。
- ダミーデータ：5医療機関 / 12医師（責任・分担、うち4名は外字）/ 6 CRC・事務局 / 4 IRB / 届出者 / 3シリーズ / 9届。
- 実装網羅：`cr_notification` / `cr_studydrug` / `cr_site` / `cr_reference` / 各マスタの列を入力UIに反映（自動採番・計算・system列を除く）。
- **要確認19件**（設計未確定）は UI に **⚠要確認** バッジで明示。30日調査対応被験薬区分は手引き未確定のため、届出区分とは切り離した暫定選択肢（要確認）で描画。

## 8. デザイン（テーマ・配色モード・折りたたみ）

- **標準（Enterprise）** — navy サイドバー・青アクセント・フラットカード。
- **モダン（Modern）** — 白サイドバー・インディゴ／ヴァイオレット・角丸・柔らかい影（モーダル/トーストにも適用）。右上トグルで切替、`localStorage` 保持。
- **ダーク／ライトモード** — デザインテーマとは独立した配色軸。中立トークン（背景・サーフェス・文字・境界・影）と実サーフェス（`--surface`）のみを `.app.mode-dark` で上書きし、各テーマのアクセント色は維持（標準＝青、モダン＝インディゴ）。右上の 🌙/☀ トグルで切替、`localStorage` 保持。
- **左メニュー折りたたみ** — サイドバー上部のシェブロンで折りたたみ／展開。折りたたむと 64px のアイコンのみのレールになり（ラベル非表示・ツールチップ表示）、コンテンツ幅が拡張。状態は `localStorage` 保持。

## 9. 品質保証

- **型チェック**（`tsc --noEmit`）/ **ビルド**（`vite build`）/ **単体テスト**（Vitest 34件）常時グリーン。
- **ブラウザE2E**（Playwright）で各画面・シナリオを実操作検証（自己完結HTMLの単体描画0エラーも確認）。
- **多エージェント敵対的レビュー**を各回実施し、確認済みの実バグを修正（例：詳細タブ化のJSX整合、モダンテーマのモーダル適用、一覧ソート順、取り込み行IDの衝突、選択肢流用、必須マーク不整合）。

## 10. 要確認・制限事項

- **デモ用サブセットXSD** — XML検証は公式XSD（`iykckn_all_v3_0_0.xsd`）ではなく要素定義から起こしたサブセット。公式XSD提供時に差し替え可能なよう分離。
- 「要確認」19件は本番で手引き（2024年3月版）との突合が必要。
- 認証（Entra ID）はユーザー切替で模擬。電子署名・SharePoint・PMDAゲートウェイ連携・監査の改ざん防止性はデモ対象外。

## 11. 本番移行（Dataverse）

`pac code add-data-source -a dataverse -t cr_notification`（他13テーブル）で型付きサービスを生成し、`dataverseRepository.ts` を実装に置換。採番・区分・職務分離・提出ゲート・外字・XML生成は**サーバー側プラグイン**へ載せ替え（`logic.ts` はライブラリとして再利用可）。切替は `VITE_USE_DATAVERSE=true` のみ。
