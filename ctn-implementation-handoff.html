CTN 治験届システム化 — 実装ハンドオフ仕様

Power Platform（Dataverse / Power Apps / Power Automate）＋ Azure Functions で実装するメンバー向けの「何をどう作るか」仕様です。

0. 読み方・前提

前提

読み手は Power Automate / Dataverse の構築経験者。本書はPAの基礎は扱わず、この案件固有で経験から推測できない部分（CTNドメイン規則・データモデル・Cat5のI/F契約・受入基準）に集中します。

本書は単独で完結させず、以下の「正本」を参照します。実装すべき項目定義はマッピング表・種別マトリクスが正本です。

正本資料役割
検討まとめ設計判断の背景（データモデル・順序番号2類型・外字）
XSDマッピング表138要素 × ラベル × STATUS × 順序番号型（XML生成の入力仕様）
届出種別マトリクス計画/変更/終了/中止/開発中止で各項目をどう扱うか（記入/継承/3.8固定NONE）
outputs/ctn-xsd-mapping/*.json上記の機械可読版（生成器・テストの入力に使用）
規制前提: 提出物の正本は XML（公式XSD iykckn_all_v3_0_0.xsd / noNamespace / UTF-8、ルート <CLINTRIALPLANNOTE>）。要素名・出現順・型・桁は最終的に PMDA配布の公式XSD と突合のこと。マッピング表はPDF抽出由来のたたき台。

1. スコープと Cat4/Cat5 境界

要点

大半は設定（Cat4）で作る。PAで作れない2点だけがコード（Cat5）：XSD検証付きXML生成と届スコープ採番。ここの切り分けが工数とCSV負荷を決める。

レイヤー技術区分
データ（マスタ・届・監査）DataverseCat 4
UI（フォーム/ビュー＋特殊UIのみPCF）モデル駆動アプリ / PCFCat 4 ＋ PCFはCat 5
ワークフロー（起票→承認→提出）Power Automate / BPFCat 4
文書保管SharePointCat 4
XML生成＋XSD検証Azure Functions (.NET)Cat 5
届スコープ採番・複雑判定Dataverse プラグイン (C#)Cat 5

段階導入： Phase 1＝社内ワークフロー電子化（設定中心・検証軽量）、Phase 2＝XML自動生成・提出（Cat5・検証重）。Cat5が入る瞬間にCSV負荷が跳ねるため、この順序。

2. Dataverse スキーマ

要点

治験成分シリーズ＝親、各届＝子。繰り返し要素は子テーブルで持ち、順序番号は型別に採番（→ §5）。物理削除は禁止、人の同一性は不変の医師IDが担う。

2.3 キー・採番・削除ポリシー

  • 届出回数（notification）＝シリーズ内連番。プラグインで採番（→ §5）。開発中止届は不要（"00"）。
  • 順序番号（治験薬）＝突合キー型：計画→終了/中止まで継承・不変、削除は欠番許容（再連番禁止）。
  • 順序番号（医師イベント）＝イベント行型：届ごとに1から採番。同一性は順序番号でなく医師マスタの不変ID。
  • 代替キーを採番衝突検出に使用（例: notification内の順序番号、series内の届出回数）。
  • 削除：親リレーションは Restrict で物理削除抑止。マスタは statecode 論理削除。医師の「削除」は削除イベント行として残す。
  • 日付：規制上の日付は「日付のみ・タイムゾーン非依存」。

3. Power Automate フロー仕様

要点

経験者向けに仕様のみ（作り方は割愛）。承認は30日超を想定し Dataverse永続化＋2フロー構成（依頼／応答処理）。承認前は提出不可をシステムで強制。

状態モデル（ctn_notification.状態）

下書き→入力完了→一次レビュー→薬事確認→承認待ち→承認済み→出力準備→提出済み
フロートリガー処理概要
A. 届起票手動／シリーズから届出種別を選択し下書き作成。種別は CLASSNOTE に設定。変更/終了/中止は最新届のデータを引き継いで複製（§6マトリクスの「継承」列）。
B. 変更届候補生成Dataverse: マスタ行の更新差分検出→該当項目を ctn_changeitem 化（STATUS=UPDATE＋変更日・理由）→ 30日起算で期限候補→ Teams/Outlook通知。3.8固定項目は除外。
C. レビュー・承認状態遷移／承認応答承認(v2)＋Dataverse永続化の2フロー。職務分離（起案者≠承認者）を強制。電子署名を記録に紐付け。
D. 提出準備状態=出力準備XML生成Functionを呼ぶ（§4）→XSD検証→エラーを画面の項目別に表示／添付不足チェック→提出パッケージ化（PDF＋XML＋添付＋命名規則）。
HITL→HOTL: 初期は全項目HITL（人が確認）。XML整形・XSD検証・コード引き・継承(NONE)の値繰越など決定的処理から順にHOTL（自動）へ。低信頼抽出・新規マスタ登録・STATUS/変更理由の妥当性はHITL維持。

4. Cat5-A: XML生成＋XSD検証 Function

要点

PA標準でXSD検証はできない。Azure Functions(.NET)で「マッピング表駆動のXMLシリアライズ＋公式XSD検証」を実装し、エラーを項目キーに逆マップして返す。生成自体は均一構造なので軽い。

I/F 契約

// POST /api/generate-ctn-xml
Request  { "notificationId": "<guid>" }
Response {
  "xml": "<?xml version=\"1.0\" encoding=\"utf-8\"?><CLINTRIALPLANNOTE ...>...",
  "valid": false,
  "issues": [
    { "severity":"Error", "xpath":"/CLINTRIALPLANNOTE/PHASECLINTRIAL",
      "itemKey":"PHASECLINTRIAL", "message":"必須項目が空です" }
  ]
}

生成（マッピング表駆動）

  • 入力は ctn_notification＋子行。マッピングJSONの順序・要素名・STATUS規則に従い直列化。
  • 各項目は <要素 STATUS="..."><VARIABLELABEL>…</VARIABLELABEL>[<CHANGEDATE>…][<CHANGEREASON>…]値</要素>。子要素は値より前。
  • 種別は 種別マトリクスで「記入/継承/不使用」を判定。3.8固定はSTATUS=NONE。
  • コード値変換（剤形・投与経路 等）は ctn_code 参照。

検証（.NET標準）

var schemas = new XmlSchemaSet { XmlResolver = null };   // 外部解決を無効化
schemas.Add(null, "iykckn_all_v3_0_0.xsd");             // noNamespace → 第1引数 null
var settings = new XmlReaderSettings {
    ValidationType = ValidationType.Schema, Schemas = schemas,
    DtdProcessing = DtdProcessing.Prohibit };           // XXE/DoS対策
settings.ValidationEventHandler += (s,e) => issues.Add(Map(e));
using var r = XmlReader.Create(xmlStream, settings);
while (r.Read()) { }                                    // 全件走査しエラー収集
  • 必須/型/桁/コード値/出現順序に加え xs:key/keyref/unique の参照整合も検証対象。
  • エラーは XPath→項目キー（XSD要素）に逆マップし、画面の出力前チェックに項目別表示。
  • セキュリティ：公式XSDは構成管理。XmlResolver=null／DtdProcessing.Prohibit。
文字ガード（同居推奨）: NFC正規化・半角/全角ルール・バイト長（文字数≠バイト数）・3.10の文字数超過時「（添付資料名）のとおり」スピル判定を、検証前段で行う。

5. Cat5-B: 採番プラグイン

要点

Dataverse標準オートナンバーはテーブル全体で一意のため「届/シリーズ・スコープの1,2,3…」に使えない。同期プラグイン(PreOperation Create)で型別に採番し、同時実行衝突に備える。

突合キー型（治験使用薬の順序番号）

// PreOperation Create on ctn_useddrug
if (変更/終了/中止届 && 前届に対応行あり)
    順序番号 = 前届の対応行の順序番号;      // 継承・不変
else
    順序番号 = max(同シリーズの順序番号) + 1; // 新規のみ加番
// 削除は欠番許容：再連番しない（突合が壊れるため）

イベント行型（医師異動の順序番号）

// PreOperation Create on ctn_doctorevent
順序番号 = max(同一届内の順序番号) + 1;   // 届ごとに1から
// 同一人物は順序番号でなく ctn_doctor の不変ID で識別
// 「追加」「削除」は同一届で別行（医師IDは同一）になり得る

同時実行制御

  • 採番キーに代替キー（一意制約）を設定し、重複時は catch→再採番リトライ。
  • 重い処理はプラグインに置かない（実行時間目安2分）。XML生成はFunctions側へ。
  • 届出区分判定・30日判定・外字正規化など複雑ロジックもサーバー側に正本化（PCF/クライアントに依存させない）。

6. 実装の入力仕様（参照）

項目レベルの実装仕様は次の2つが正本です。コード/フロー/Functionはこれを入力として実装してください。

  • XSDマッピング表 … 138要素 × ラベル × 取りうるSTATUS × 順序番号型（出現セクション付き）。機械可読は outputs/ctn-xsd-mapping/ctn-xsd-mapping.json。
  • 届出種別マトリクス … 計画/変更/終了/中止/開発中止 × 各項目（記入/継承/△/—、3.8固定NONE）。機械可読は outputs/ctn-xsd-mapping/ctn-notification-matrix.json。
種別の作り分け（再掲）: 種別は STATUS ではなく 届出分類(CLASSNOTE) で指定。終了/中止の実績（中止情報・数量・被験者数）は3.8によりSTATUS=NONEで記入。開発中止届は抜粋フォーマット（22項目）で引き継ぎではない。

7. ハマりどころ（先回り）

論点問題対応
採番標準オートナンバーはテーブル全体一意同期プラグインでスコープ採番（§5）
文字数列長=文字数だがPMDA制限はバイト数（ファイル名255byte、カナ氏名 全角50/半角100）。サロゲートペア注意バイト換算の検証ロジックを別実装（§4）
日付User Local 既定だと届出年月日が1日ずれる「日付のみ・TZ非依存」設定
集計ロールアップ列は非同期（即時反映されない）即時性が要る集計は数式列/プラグインで補う
削除物理削除で履歴・突合が崩れる親はRestrict、マスタは論理削除、医師削除はイベント行で残す
PCFクライアント検証は迂回され得る／Cat5で検証文書が重いPCFはUX向上のみ、同等の正本をサーバー側に
ALMPower Platformは月次更新全部品をソリューション化、開発/検証/本番分離、本番はマネージド、回帰テストパックで再検証判断

8. 受入基準（OQ）とテストデータ

要点

「人が最終承認するからCSV不要」は不可。人間の最終承認をコントロールとして登録し、リスクベースで検証深度を配分。下記8項目はOQで必ず通す。

  1. 承認前は提出に進めない（HITLの構造的強制）
  2. 監査証跡が記録され、一般ユーザーが無効化・改変できない
  3. 職務分離（起案者≠承認者）が強制される
  4. 電子署名が記録に紐づく
  5. XMLがXSDスキーマ検証を通過する
  6. 外字が検出・置換・記録され、説明資料が生成される
  7. 治験薬の順序番号が終了届まで保持され、数量を突合できる（突合キー型）
  8. 分担医師の追加・削除がイベント行（番号＋区分＋日付＋理由）で正しく生成される（イベント行型）

テストデータ例（医師異動・イベント行型）

届順序番号区分医師ID氏名
初回計画届1 / 2登録D001 / D002佐藤 / 鈴木
第1回変更1 / 2追加 / 削除D003 / D001田中追加・佐藤削除
第2回変更1〜3追加/追加/削除D004 / D005 / D005同一届でD005が追加＋削除の2行
第3回変更1変更D002鈴木→高橋（IDは不変）

検証観点：番号は届ごとに振り直し／同一届で追加＋削除の2行可／同一性はIDが担保。詳細は 検討まとめ Section 04。
